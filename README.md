## סקירה

המערכת מקליטה את מסך המשתמש, מחלקת את ההקלטה לקטעים (chunks) של 30 שניות, ומעלה אותם לשרת ברקע. כל קטע נשמר מקומית לפני ההעלאה, כך שההקלטה שורדת ניתוקי רשת, ריענון דף, וקריסות דפדפן. בסיום ההקלטה, השרת מאחד את כל הקטעים לסרטון אחד שלם ונגין, המאוחסן לפי מזהה ייחודי.

**עקרון מנחה:** ההקלטה לעולם לא נפגעת מבעיות רשת. ההעלאה היא תהליך נפרד שמתקדם בקצב שלו, ומדביק כשיש חיבור.

**יכולות עיקריות:**
- הקלטת מסך עם חלוקה לקטעים עצמאיים כל 30 שניות
- אחסון מקומי ב-IndexedDB — עמידות לריענון וקריסה
- העלאה ברקע עם ניסיונות חוזרים אוטומטיים
- איחוד קטעים לסרטון אחד באמצעות ffmpeg
- מזהה ייחודי לכל הקלטה לצורך מעקב ואחזור
- ממשק עם חיווי חיבור, סרגל התקדמות, ולוגים בזמן אמת

## ארכיטקטורה

המערכת בנויה בשלוש שכבות מופרדות, כל אחת עם אחריות אחת:

```
┌─────────────────────────────────────────────────────────┐
│                      דפדפן (Client)                      │
│                                                          │
│  ┌────────────┐    ┌──────────────┐    ┌─────────────┐  │
│  │  Recorder  │───▶│  IndexedDB   │◀───│  Uploader   │  │
│  │  (הקלטה)   │    │  (אחסון)     │    │  (העלאה)    │  │
│  └────────────┘    └──────────────┘    └──────┬──────┘  │
│   מקליט מסך          שומר chunks         מעלה ברקע      │
│   כל 30 שניות        עם סטטוס            כשיש חיבור      │
└──────────────────────────────────────────────┼─────────┘
                                                │ HTTP
                                                ▼
┌─────────────────────────────────────────────────────────┐
│                    Mock Server (Node)                    │
│                                                          │
│   מקבל chunks ──▶ שומר בדיסק ──▶ מאחד (ffmpeg) ──▶       │
│   /upload         storage/         completed/rec_ID.webm │
└─────────────────────────────────────────────────────────┘
```

### שכבה 1 — Recorder (`src/recorder.ts`)
אחראית **רק** על הקלטה. משתמשת ב-`getDisplayMedia` לתפיסת המסך וב-`MediaRecorder` להקלטה. כל 30 שניות מפעילה מחדש את ה-MediaRecorder, כך שכל קטע הוא קובץ webm עצמאי ותקין. שומרת כל קטע ל-IndexedDB. **אינה מכירה את השרת** — ההקלטה ממשיכה גם ללא חיבור.

### שכבה 2 — Storage (`src/db.ts`)
עטיפה ל-IndexedDB, מסד הנתונים המקומי של הדפדפן. שומרת כל קטע עם סטטוס (`uploaded`), ומאפשרת שליפה של קטעים ממתינים. **שורדת ריענון דף וקריסת דפדפן** — כאן טמונה העמידות של המערכת.

### שכבה 3 — Uploader (`src/uploader.ts`)
רצה ברקע בלולאה מתמדת. שולפת קטעים שעדיין לא הועלו, מעלה אותם לשרת, ומסמנת כמועלים. אם העלאה נכשלת — הקטע נשאר במסד וינוסה בסבב הבא. **אין הגבלה על משך הניתוק** — כשהחיבור חוזר, כל הקטעים הממתינים עולים.

### השרת (`mock-server.ts`)
מקבל קטעים, שומר בדיסק, ובסיום מאחד את כולם לסרטון אחד באמצעות ffmpeg.

---

## זרימת הנתונים

1. המשתמש מתחיל הקלטה → נוצר `recordingId` ייחודי
2. כל 30 שניות → נוצר קטע → נשמר ב-IndexedDB (`uploaded: false`)
3. ה-Uploader ברקע → שולף קטעים ממתינים → מעלה → מסמן `uploaded: true`
4. אם אין רשת → הקטע נשאר ממתין, ינוסה שוב
5. בסיום → המערכת מוודאת שכל הקטעים עלו → מודיעה לשרת
6. השרת → מאחד את כל הקטעים של אותו `recordingId` לסרטון אחד → שומר ב-`completed/`

## הרצה

### דרישות מקדימות
- Node.js (גרסה 18 ומעלה)
- ffmpeg (לאיחוד הסרטונים)

### התקנה
```bash
npm install
```

### הרצת השרת
```bash
npm run mock-server
```
השרת יעלה על `http://localhost:4000`.

### פתיחת הממשק
בדפדפן, גש אל:
```
http://localhost:4000/index.html
```

### בנייה (קימפול TypeScript ל-JavaScript)
```bash
npm run build
```

- שרידות הנתונים בשליפה חוזרת

## החלטות תכנון

**הפרדת הקלטה מהעלאה.** ההחלטה המרכזית. במקום שההקלטה תשלח ישירות לשרת, היא שומרת מקומית, וההעלאה היא תהליך נפרד. כך כשל ברשת לעולם לא פוגע בהקלטה עצמה. זה חיוני בהקלטות ארוכות — העלאת הקלטה של שעתיים לוקחת זמן, וההקלטה חייבת להימשך ללא תלות בכך.

**IndexedDB ולא זיכרון בלבד.** בחירה ב-IndexedDB (ולא במערך בזיכרון) נותנת עמידות לריענון דף ולקריסת דפדפן. הקטעים שורדים גם אם הטאב נסגר.

**קטעים עצמאיים.** ה-MediaRecorder מופעל מחדש כל 30 שניות, כך שכל קטע הוא קובץ webm שלם עם כותרת משלו — וניתן לנגן כל קטע בנפרד.

**איחוד עם ffmpeg.** חיבור בייטים פשוט של קבצי webm יוצר כותרות כפולות שנגנים אינם יודעים לנגן. ffmpeg מאחד את הזרמים ברמת הקודק ומייצר קובץ תקין אחד. זהו הכלי התעשייתי הסטנדרטי למשימה.

**מזהה ייחודי לכל הקלטה.** ה-`recordingId` מקשר את כל הקטעים לסרטון המאוחד הנכון. כאשר לקוח רוצה לצפות בהקלטה, נשלח סרטון אחד שלם — לא עשרות קטעים.

**שמירת הקטעים המקוריים.** גם לאחר האיחוד, הקטעים הבודדים נשמרים כגיבוי, לשחזור או איחוד מחדש.

## טיפול בכשלים

המערכת מתמודדת עם שלושה תרחישי כשל:

**ניתוק רשת זמני.** ה-Uploader מנסה שוב אוטומטית. הקטע נשאר במסד עד שההעלאה מצליחה.

**ניתוק ממושך.** אין הגבלה על משך הניתוק. הקטעים מצטברים ב-IndexedDB, וכשהחיבור חוזר — כולם עולים בזה אחר זה.

**קריסת דפדפן / ריענון.** הקטעים שמורים ב-IndexedDB ושורדים. עם טעינת הדף מחדש, ה-Uploader ממשיך להעלות את מה שנותר.

לוגים בשלוש רמות (הצלחה / אזהרה / שגיאה) מציגים את מצב המערכת בזמן אמת. חיווי חיבור וסרגל התקדמות מספקים משוב ויזואלי מיידי.

## מגבלות ושיפורים עתידיים

מתוך שקיפות, אלו הנקודות שהייתי משפר בגרסת production:

- **ניקוי IndexedDB.** כרגע הקטעים נשארים במסד המקומי גם לאחר העלאה מוצלחת. בהמשך הייתי מוחק אותם לאחר אישור העלאה, לניהול זיכרון בצד הלקוח.
- **העלאה מקבילית.** ההעלאה כיום סדרתית (קטע אחרי קטע). לאחר ניתוק ארוך עם קטעים רבים, העלאה מקבילית עם הגבלת מקביליות תדביק מהר יותר.
- **אחסון בענן.** במקום שמירה בדיסק מקומי, אחסון אובייקטים (כמו S3) עם מטא-דאטה ב-DB.
- **ולידציה ואבטחה בשרת.** אימות גודל וסוג הקטעים, הגבלת קצב, ואימות זהות המשתמש.
- **אימות שלמות.** checksum לכל קטע כדי לוודא שהגיע שלם לשרת.
